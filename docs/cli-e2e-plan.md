# CLI E2E Plan

This is the next implementation plan for getting a real agent/CLI request through the deployed PWA approval flow.

## Current State

- Web/PWA lives in `apps/web`.
- Deployed app: `https://sickrat.dev`.
- Worker already supports:
  - Cloudflare OAuth login/provisioning prototype.
  - D1-backed push subscriptions.
  - D1-backed approval requests.
  - PWA push notifications.
  - Durable Object realtime channel for foreground approvals.
  - Passkey PRF-protected local vault key in the PWA.
  - Client-side encrypted secret creation and metadata listing.
- Current approval requests are still demo-generated by `/api/push/test`.
- Next goal: replace demo requests with real CLI-created requests.

## CLI Choice

Use a Bun/TypeScript CLI for the first working loop.

Reasons:

- Faster iteration than Rust right now.
- Can share TypeScript protocol types with the PWA/Worker.
- Native `fetch`, WebSocket, JSON, and process spawning are enough for the first milestone.
- Rust can come later after the protocol is proven, especially for static binaries and stronger memory handling.

Proposed package:

```text
packages/protocol/        shared request/response schemas
apps/cli/                 Bun CLI
```

Root scripts later:

```json
{
  "cli": "bun run apps/cli/src/index.ts"
}
```

## Device Pairing Purpose

Pairing answers: which machines are allowed to ask this phone/PWA for secrets?

It provides:

- Device identity: `mac-mini.local`, not anonymous HTTP.
- Request authorization: Worker rejects unpaired/revoked devices.
- Encryption target: PWA can trust a paired device public key, or an ephemeral key signed by that device.
- Revocation: user can remove lost/stale machines.
- Audit: approvals can show device, command, refs, time.
- UX trust: approval screen can say "Mac mini, paired yesterday".

## Phase 1: Minimal Working E2E

Status on 2026-06-07: implemented and deployed for the prototype Worker.

Implemented pieces:

- `apps/cli` Bun/TypeScript CLI with `pair` and `request`.
- `packages/protocol` shared canonical approval payload and envelope types.
- Pairing code APIs and PWA Pair CLI screen.
- Signed CLI approval request API.
- PWA approval decrypts requested ciphertext locally and returns an ECDH/HKDF/AES-GCM encrypted grant.
- CLI polls the grant endpoint, decrypts the grant, and writes the requested secret value to stdout.

Prototype caveats:

- Requests notify the most recent registered push subscription because account/vault ownership is not fully modeled yet.
- Bun is the intended runtime, but this Mac currently does not have `bun`; use `npm run cli:node -- ...` as the local fallback with Node 26.
- `run --env-file` is still Phase 2.

This phase prioritizes the product loop over perfect crypto.

Flow:

1. User stores a secret in the PWA:
   - Open `/?screen=secrets`.
   - Create/unlock passkey-protected vault key.
   - Add `openai/api-key`.
   - Secret value is encrypted locally and ciphertext is stored in D1.

2. CLI pairs:
   - `sickrat pair https://sickrat.dev`
   - CLI generates a device keypair.
   - CLI asks Worker for a short pairing code.
   - PWA has a Pair Device screen where user enters/approves the code.
   - Worker stores device id, label, public key, created timestamp, revoked flag.
   - CLI stores worker URL, device id, private key locally.

3. CLI requests a secret:
   - `sickrat request openai/api-key --message "Run the smoke test against the real API"`
   - CLI generates an ephemeral grant keypair.
   - CLI signs request, including the message, with device private key.
   - Worker verifies paired device and creates approval request.
   - Worker publishes request over Durable Object realtime channel and sends Web Push.

4. PWA approves:
   - Approval screen shows device, command, refs, timestamp.
   - User unlocks vault if needed.
   - PWA decrypts requested secret locally.
   - PWA encrypts plaintext to CLI ephemeral public key.
   - PWA POSTs encrypted grant to Worker.

5. CLI receives grant:
   - CLI waits/polls for grant.
   - CLI decrypts with ephemeral private key.
   - CLI prints value for `request` command only in first dev loop.

Deliverable for Phase 1:

```sh
sickrat pair https://sickrat.dev
sickrat request openai/api-key --message "Run the smoke test against the real API"
```

Expected result: phone/PWA approval returns encrypted value to CLI, CLI decrypts and writes the requested value to stdout.

## Phase 2: `run --env-file`

Add:

```sh
sickrat run --env-file .env -- npm test
```

Behavior:

- Read `.env`.
- Detect values shaped like `sickrat://...`.
- Create one approval request containing all refs, command, cwd, host, and process argv.
- Receive grant map.
- Spawn child process with resolved env vars.
- Do not write plaintext secrets to disk.
- Redact secrets from CLI logs.

Example `.env`:

```env
OPENAI_API_KEY=sickrat://default/openai/api-key
DATABASE_URL=sickrat://prod/database/url
```

## Worker Additions

### D1 Schema

Add tables:

```sql
CREATE TABLE IF NOT EXISTS devices (
  id TEXT PRIMARY KEY,
  label TEXT NOT NULL,
  public_key TEXT NOT NULL,
  created_at TEXT NOT NULL,
  revoked_at TEXT
);

CREATE TABLE IF NOT EXISTS pairing_codes (
  code TEXT PRIMARY KEY,
  device_id TEXT NOT NULL,
  label TEXT NOT NULL,
  public_key TEXT NOT NULL,
  expires_at TEXT NOT NULL,
  approved_at TEXT
);

ALTER TABLE approval_requests ADD COLUMN device_id TEXT;
ALTER TABLE approval_requests ADD COLUMN ephemeral_public_key TEXT;
ALTER TABLE approval_requests ADD COLUMN grant_ciphertext TEXT;
ALTER TABLE approval_requests ADD COLUMN grant_ready_at TEXT;
```

For prototype speed, `ensureSchema` can keep doing lazy schema changes. Before production, move to D1 migrations.

### Pairing APIs

```http
POST /api/devices/pairing-codes
```

Request:

```json
{
  "label": "mac-mini.local",
  "publicKey": "base64url..."
}
```

Response:

```json
{
  "code": "482913",
  "deviceId": "uuid",
  "expiresAt": "iso"
}
```

```http
GET /api/devices/pairing-codes/:code
POST /api/devices/pairing-codes/:code/approve
```

Approval creates `devices` row and marks pairing code approved.

```http
GET /api/devices/pairing-codes/:code/status
```

CLI polls this until approved/expired.

### Approval Request APIs

```http
POST /api/approval-requests
```

Request:

```json
{
  "deviceId": "uuid",
  "command": "sickrat request openai/api-key",
  "message": "Run the smoke test against the real API",
  "secretRefs": ["openai/api-key"],
  "ephemeralPublicKey": "base64url...",
  "signature": "base64url..."
}
```

Response:

```json
{
  "requestId": "uuid",
  "status": "pending"
}
```

Worker verifies device signature, creates approval request, sends realtime + push.

```http
POST /api/approvals/:id/grant
```

PWA posts encrypted grant:

```json
{
  "grantCiphertext": "base64url..."
}
```

```http
GET /api/approval-requests/:id/grant
```

CLI polls until grant is ready, denied, expired, or timed out.


### Approval Grant Logic

On approve:

1. Ensure vault key is unlocked.
2. Fetch encrypted secret records for requested refs.
3. Decrypt with vault key.
4. Encrypt grant map for CLI ephemeral public key.
5. POST grant to Worker.
6. Mark approval approved.

The current `/api/secrets` only lists metadata. Add a fetch-by-ref endpoint that returns ciphertext for the PWA:

```http
POST /api/secrets/resolve
```

Request:

```json
{
  "refs": ["openai/api-key"]
}
```

Response includes ciphertext, iv, salt/kdf metadata, never plaintext.

## Crypto Plan

Prototype:

- Device signing key: Ed25519 if available in Bun/WebCrypto; otherwise use P-256 ECDSA for browser compatibility.
- Grant encryption: X25519 would be ideal, but browser support may be uneven. For first loop, use P-256 ECDH + HKDF + AES-GCM because WebCrypto supports it broadly.

Grant envelope:

```json
{
  "alg": "P-256-ECDH-HKDF-SHA256-AES-256-GCM",
  "senderPublicKey": "base64url...",
  "iv": "base64url...",
  "ciphertext": "base64url..."
}
```

Plaintext grant map before encryption:

```json
{
  "requestId": "uuid",
  "secrets": {
    "openai/api-key": "plaintext"
  },
  "expiresAt": "iso"
}
```

## Known Non-Goals For First Loop

- Full 1Password CLI compatibility.
- Perfect memory zeroization.
- Multi-device vault recovery.
- Public Cloudflare OAuth client review.
- Production D1 migrations.
- React Router cleanup.
- Native mobile app.
